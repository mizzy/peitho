# PDF export: flatten gradients at print time (Issue #142)

## Summary

PDFs produced by `peitho export pdf`, when opened in macOS Preview.app, start entirely white and take several to a dozen seconds to fill in each page. The root cause is **CSS gradients**: Chrome's Skia backend emits them as Type 4 shadings (PostScript-style functional shading) + soft masks, and CoreGraphics's per-pixel function evaluation is pathologically slow. The fix is to embed a script in the print-time `pdf.html` that **has Chrome itself rasterize just the gradient backgrounds into data-URL images**. Text remains fully vector (selectable and searchable).

## Measured facts (2026-07-06, no need to re-investigate)

Measured on a real deck (decks/carina, 11 slides, 1.73MB). Benchmark uses PDFKit (the same renderer as Preview.app) rendering offscreen:

| Metric | Current (vector shadings) | Flatten prototype |
|---|---|---|
| First page (starry cover) | 14,204ms | 29ms (~490x) |
| All 11 pages total | 15,109ms | 281ms |
| `/Shading` objects | 19 | 0 |
| `/SMask` | 9 | 0 |
| Embedded images | 0 | 2 / 789KB |

- **The slowness is entirely from gradients.** A control experiment that replaced only gradients with flat colors ran at 5.8ms/page (0 shadings) even with 40 Type 3 fonts remaining. Fonts are effectively irrelevant to speed
- Type 4 shadings are not just slow in Preview-family renderers, they are **rendered incorrectly** (stars drawn with radial-gradient nearly disappear). The flattened version is more faithful to what's on screen
- When multiple slides share the same gradient background, Skia deduplicates the identical data URLs, so the image count is effectively one (10 shared slides ended up as only 2 objects)
- Confirmed in the prototype that printing after script completion works under `--virtual-time-budget` (Chrome 149)
- Side finding (**out of scope for this PR**, separate root cause): 1.37MB of the 1.7MB PDF is a whole-font embed of Osaka-Mono. Code elements without a monospace declaration fall to Chrome's default monospace font (Osaka-Mono on Japanese locale), and Skia fails to subset it. The built-in theme's `.slot-code` has the same hole → handled in a separate Issue

## Design decisions

- **Default behavior with no flag** (zero-config policy). No case measured where it regresses: visually more faithful, text stays vector, size increase (+0.77MB) is within budget and will drop below current after the Osaka-Mono problem is fixed
- **Have Chrome itself rasterize** (SVG `foreignObject` → `<img>` → canvas → `toDataURL`). We do not adopt the alternative of parsing and drawing CSS gradient syntax on the Rust side (tiny-skia etc.) — that would require reimplementing the whole `background-position` / `size` / multi-layer background / conic compatibility matrix. Using Chrome's actual rendering guarantees zero interpretation drift by construction
- The script is inlined into the `pdf.html` generated by `render_pdf_document`. The JS asset lives at `crates/peitho-core/src/pdf_flatten.js` and is pulled in via `include_str!` (source itself, not a build artifact like shell.js; not part of the TS build chain — it is a small print-only helper unrelated to the §16 event contract)
- Capture resolution is fixed at **2x the element's CSS pixels**. 2560px for a 1280px-wide slide — 1.33x oversampling for a 1920px output page. Gradients are low frequency, so this is plenty for print (verified by eye in the prototype)
- Targets are **actual elements and `::before` / `::after` pseudo-elements** whose computed `background-image` contains `gradient(`. Pseudo-elements' element-side style cannot be directly rewritten, so replace them via a generated class + `!important` rule injection. If a pseudo-element's dimensions cannot be resolved to px, skip that pseudo-element (fall back to leaving it as vector = slow but correct)
- Do not flatten targets where a faithful capture cannot be guaranteed: mixed `url()` layers, `background-attachment: fixed`, `background-clip: text`, non-`normal` `background-blend-mode`, `background-color` that isn't fully transparent + non-`border-box` clip + non-zero inset, inline elements split across multiple fragments (multiple `getClientRects`), element dimensions that can't be resolved to px, pseudo-elements with padding/border, and huge backgrounds that exceed Chrome's canvas ceiling. Skip these and leave the original vector gradient
- Wait for `document.fonts.ready` and `load` completion before collecting targets, so that final geometry (post-reflow after webfonts and images) is captured (`--virtual-time-budget` accommodates this wait)
- Wrap each element in `try/catch`; on failure, leave the original gradient (**degradation direction is always "slow but correct"** — never produce a white PDF or a missing background)
- Add `--virtual-time-budget=10000` to `run_chrome_print`. It is virtual time, so wall-clock cost on static pages is negligible (verify with E2E that gradient-free decks don't get slower to export)
- After replacement, set `background-size: {w}px {h}px` / `background-position: 0 0` / `background-repeat: no-repeat` plus explicit `background-origin: border-box` / `background-clip: border-box` so the image overlays the border-box area the raster represents exactly (also prevents the layout rules of the original multi-layer background from being misapplied to the single image)

## Non-goals

- Do not flatten `box-shadow` / `filter`. Skia already images those (shadings only remain for gradients)
- Osaka-Mono whole-font embed problem (above) is a separate Issue
- Do not introduce a full-page-rasterize option (`--rasterize`). This approach solves it while keeping text vector

### Task 2: `pdf_flatten.js` + embed into `render_pdf_document`

- Red: unit test on `render_pdf_document`'s output verifying it contains a `<script>` and the flatten script's identifiers (function names etc.). The existing assertions on slide HTML / `@page` etc. must not break
- Green: add `crates/peitho-core/src/pdf_flatten.js` and embed into the template via `include_str!`
- Script spec:
  - Walk all elements + `::before` / `::after`, collect those whose `background-image` contains `gradient(`
  - For each target: snapshot the computed `background-image` / `position` / `size` / `repeat` / `origin` / `clip` + geometry (padding / border width & style, transparent border-color), wrap the resulting cloned div in an SVG `foreignObject`, load into `<img>` via data URL, draw into a canvas at 2x, `toDataURL('image/png')`. Deliberately do not copy `background-color` (the element's own color renders under the raster as-is, preventing double compositing). Snapshotting (not holding a live `CSSStyleDeclaration`) also prevents contamination from DOM rewrites on prior targets
  - Real elements are replaced via inline style; pseudo-elements via a generated class + injected `<style>` rule (`!important`)
  - Per-element `try/catch`; on failure, no change
  - Expose completion via a `data-peitho-pdf-flattened` attribute on `document.documentElement` (for E2E / debugging). Do not use `document.title` — it becomes the PDF's `/Title` metadata
