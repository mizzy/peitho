# peitho

An HTML-native presentation tool with Markdown as the source of truth. The authoritative design reference is `docs/PEITHO_KICKOFF.md` (the kickoff spec). When in doubt about a design decision, check §18 "Undecided Items" in the spec; for new decisions not covered there, check with the author.

## Language

All written artifacts in this project must be in English. This includes: documentation (README, docs/, inline comments, commit messages), GitHub issues (titles and bodies), pull requests (titles, bodies, and review comments), and code identifiers. Do not write these in Japanese or any other language. If the author writes to you in Japanese, still produce artifacts in English.

## The three pillars (invariants that must not be broken)

1. **Separation of content and design**: Content is Markdown, design is layout HTML+CSS. Do not mix them
2. **Git-manageable HTML/CSS layouts**: The layout itself is the schema (`<slot name accepts arity>`). Do not split the contract into a separate file
3. **Type-checked slot contracts and keyed overrides**: Slot excess/deficiency, type mismatches, broken references, and unassigned content are all build errors with line numbers + help. **Silent dropping is absolutely forbidden** (never let the parser swallow unknown structures with `_ => {}`)

Other invariants:
- typestate `Parsed→Mapped→Checked→Rendered`. Phase constructors are private within the crate. An unchecked deck cannot be passed to the renderer (pinned down by a compile_fail doctest)
- Multiple layouts use hybrid dispatch (type-driven approach from §18 adopted at the author's discretion, 2026-07-03): explicit `{"layout":"name"}` > unconditional if there's only one > unique structural match. Ambiguous or zero matches are never silently resolved — they are build errors. Slides carry their own Layout from Mapped onward, so no lookup-failure path is created downstream
- Syntax highlighting is done at build time with syntect (adopted 2026-07-03). Output is spans with `hl-*` classes, colors come from the theme CSS. Custom syntaxes come from `syntaxes:` frontmatter or `syntaxes/` next to the deck and augment the built-in set. Unknown language tags are a parse-time error with a line number (no tag means plain text)
- Deck-level settings ride in YAML frontmatter at the top of the deck (author decision 2026-07-03: "zero-config, settings live in Markdown frontmatter"). Supported keys: `time` (planned presentation time, `15m`/`90s`/`1h30m`/bare integer minutes), `aspect_ratio`, `resolution` (PDF-only physical page size, `WxH` CSS px), `breaks` (single newlines in slide body Markdown render as hard line breaks when `true`), `layouts`, `css`, `syntaxes`, `fonts`, and `code_images`. For asset keys, asset resolution is explicit frontmatter > deck-adjacent auto-detect (`layouts/` / `css/` / `syntaxes/` / `fonts/`) > built-in embed/default for layouts/css/syntaxes or no extra asset for fonts; explicit non-existent paths are line-numbered build errors, not silent fallback. Asset values may be a file or a directory: layouts/css/syntaxes read `*.html`, `*.css`, and `*.sublime-syntax` in deterministic filename order, while fonts are copied verbatim without an extension filter so `.woff2`, `.ttf`, and `@font-face` CSS files can sit side by side. The frontmatter body is restricted to flat `key:` lines, plus one nested `code_images:` mapping level (trailing style blank lines allowed); unknown keys, invalid values, markdown swallowed by a missing closing `---`, a leading `---` without valid frontmatter, and settings anywhere but the top are all line-numbered build errors with help — no silent path exists. Validation happens once at `PlannedTime`'s construction (nonzero, overflow, ≤ `Number.MAX_SAFE_INTEGER` so JSON.parse can't round it), and the validated newtype rides on `Deck<P>` through every phase; milliseconds appear only at consumption boundaries (manifest). Design records: `docs/specs/2026-07-03-time-tracking-design.md`, `docs/plans/2026-07-05-frontmatter-layouts-css.md`
- code_images renderer resolution (2026-07-15, Issue #252): Mermaid is built in via `merman`; `dot`, `plantuml`, and every other diagram tag stay user-declared external commands. `CodeImagesConfig::renderer_for(tag)` is the single typed resolver (`External(&CodeImageCommand)` for explicit frontmatter, `BuiltinMermaid` for bare `mermaid`, `None` otherwise). Both parser unknown-language validation and `code_images` transformation must go through that resolver — no `tag == "mermaid"` carve-outs. `SvgRunner` remains only "run an external command"; built-in Mermaid is dispatched before the trait is consulted, and both paths converge on validate → normalize → cache. Bare `true`/`false` `code_images` values are line-numbered errors reserving future opt-out syntax; explicit `code_images.mermaid: ...` overrides the built-in renderer.
- code_images SVG intrinsic size (2026-07-12, Issue #261): every SVG written to `.peitho/code-images-cache/` carries usable absolute `width`/`height` on the root tag — mermaid-style output (`width="100%"`, no height, viewBox only) collapses to 0×0 via `<img>` otherwise. Normalization happens once at the cache-write seam (viewBox supplies missing/unusable dimensions; graphviz output passes through byte-identical); cached files lacking the invariant are cache misses (self-heal); no-dims-no-viewBox and root-tag-not-found are distinct line-numbered build errors. Attribute names match exact case (XML semantics — browsers ignore `WIDTH`/`viewbox`, so peitho must too). User-referenced `![](foo.svg)` assets stay verbatim by design. Design record: `docs/plans/2026-07-12-code-images-svg-intrinsic-size.md`
- Agenda sections (2026-07-04): a slide's page settings comment may declare `{"section":"Name","time":"1m"}` (section and time must appear together; the marked slide starts a section that runs until the next marker). Ranges are derived at parse end; if any marker exists the first slide must carry one; when frontmatter `time` is present the section total must equal it, when absent the total becomes the deck's planned time (so a second validation point exists at parse end, not only at `PlannedTime` construction). A slide accepts at most ONE page settings comment — a second one is a build error (the old field-wise last-wins merge could silently drop a section marker). Sections ride `DeckSettings` through every phase into `manifest.json` `sections` (always present, empty when unused); the presenter agenda measures actuals client-side (cumulative, in-memory). Design record: `docs/specs/2026-07-04-presenter-agenda-design.md`
- Per-slide draft/skip flags (2026-07-12, Issue #242): page settings also support `{"draft":true}` and `{"skip":true}`. Draft slides are dropped at parse end before final `ParsedSlide.index` assignment, before section resolution, and before any downstream phase can observe them; `draft` never becomes a `ParsedSlide`/Mapped/Checked/Rendered field. Derived keys are parse-time identity and are not re-derived after a draft drop; `source_index` rides Parsed→Mapped→Checked so post-parse errors still report source slide numbers. Invalid combinations are line-numbered build errors: `draft+skip`, `draft+section`, and all slides marked draft. `skip` rides the same Parsed→Mapped→Checked path as notes and surfaces as `ManifestSlide.skip` with serde default `false`; skipped slides are real slides, stay counted, keep notes, remain in PDF/publish output, and do not affect section ranges/time. Shell keyboard components still emit only request events: present opens on the first non-skipped slide, preview does so only when there is no restored session state (restored preview indexes stay exact, including skipped slides), present and preview single-mode `next`/`prev` resolve to the next non-skipped slide or no-op, while direct key/index navigation, Home/End first/last, and preview grid movement can land on skipped slides. Design record: `docs/plans/2026-07-12-per-slide-draft-skip.md`
- Speaker notes (2026-07-04): non-JSON HTML comments in a slide body are collected as that slide's speaker note (Marp / k1LoW/deck-style). JSON-vs-plaintext is the discriminator (reuses `parse_page_comment`) so page settings and notes coexist; empty comments are ignored; multiple comments per slide are joined with a blank line; position is unconstrained (before/inside/after the body). Notes ride `notes: Option<String>` on the slide through every phase and are emitted into `notes.json` (keyed by `SlideKey`, built at Rendered when keys are final). **Notes never enter `dist/`** — only `peitho present` reads `notes.json`; the publish contamination check enforces this. Presenter renders notes as `textContent` (plaintext v1); Markdown interpretation is a future extension the versioned schema leaves room for. Design record: `docs/plans/2026-07-04-speaker-notes.md`
- Presenter view (2026-07-04, PR #79 + follow-ups): state-synced timer UI with a fixed 42vh notes panel and a height-derived 16:9 preview stage that keeps left-column widths stable. Space bar maps to `playpause` (start/pause/resume derived from `TimerState`); the presenter emits `peitho:timercontrol` and the shell executes the transition. Empty notes render as a dimmed placeholder rather than a blank block. Design records: `docs/plans/2026-07-04-presenter-redesign.md`, `docs/plans/2026-07-04-presenter-redesign-followups.md`
- Display swap (2026-07-04, Issue #108): `S` key / presenter Swap button is an escape hatch for display misidentification — each window navigates (`location.replace`) to its counterpart route, so the windows stay put and only content roles swap. Role = URL via four routes / two files: `present.html`⇄`/presenter-swapped` (slides window), `/presenter`⇄`/present-swapped` (presenter window); the swapped aliases are extensionless and dot-free so Chrome placement keys stay distinct (dot pitfall below). Sync state is an **absolute** `{"swapped":bool}` (never a toggle command — the channel coalesces), the server tracks current `index`/`swapped`, and **every** `/sync` GET response (handshake and poll) carries that state, which the client replays idempotently after each message — this per-poll replay is load-bearing for convergence (a live window that misses a coalesced-away swap never reloads, so handshake-only replay is insufficient) and is what preserves the slide index across swap navigations. UI only emits `peitho:swaprequest` (§16); the sync bridge posts/navigates. The slides page gates the shortcut on `present.json` `presenterOpen` (a solo slides window must not swap itself away). Known tradeoffs: the presenter timer resets on swap; post-swap slides sit windowed (press `f`). Chord-modified keys (meta/ctrl/alt) are ignored by all shortcuts via a shared `hasChordModifier` guard (Cmd+S/Cmd+F keep their browser meaning). Design record: `docs/plans/2026-07-04-display-swap.md`
- Preview loop (2026-07-07, Issue #170): `peitho preview` writes only `.peitho/preview-cache/build-<generation>/`, serves `index.html` plus `preview.js`, and watches the same deck/asset roots as `build --watch` without watching the preview cache. Each successful watch rebuild emits a fresh generation directory, swaps the server root only after emit completes, keeps the immediately previous generation for in-flight requests, prunes older generations, and increments the server-owned `/sync` `generation` absolute state (no transient reload message). Every `/sync` GET response (handshake and 200 poll) carries `generation`; the preview client handshakes before fetching content and saves `{mode,index}` in `sessionStorage` before `location.reload()` when generation changes. If the first build fails, preview still starts with an escaped error page that polls generation and reloads on the next successful build. Preview keyboard handling follows §16: shortcuts only emit request events (`peitho:overviewrequest` / `peitho:navigate`), all transitions live in the preview shell, and every shortcut goes through `hasChordModifier`.
- Single source for the contract: domain types are authoritative in peitho-core (Rust). TS types are generated into `bindings/*.ts` via ts-rs and committed. CI checks for drift
- §16 event contract: only the shell executes transitions. UI components only emit request events like `peitho:navigate`/`peitho:timercontrol`. The slide body itself has no knowledge of the shell's existence
- Do not mix the presentation shell or notes into distributed artifacts (dist/) (publish gatekeeps this with a contamination check)

## Structure

```
crates/peitho-core/   Contract & pipeline (parser/layout/mapping/check/render/theme/manifest/notes)
crates/peitho/        CLI (new/build/layouts/doctor/preview/present/publish/export/completions), server.rs (serving + /sync long polling), browser.rs, displays.rs
packages/peitho-present/  TS presentation shell (canvas/shell/controls/keyboard/sync/presenter)
bindings/             ts-rs generated TS types (committed)
layouts/ themes/ examples/  Shared layouts, base theme, samples. The default layouts/base.css is embedded and used only after explicit frontmatter and deck-adjacent auto-detect have both been checked. The presentation shell dist/shell.js and preview shell dist/preview.js are embedded, committed, and checked for drift; `--shell` remains the only CLI-side present-shell bundle swap and is a dev/debug override. `layouts:` / `css:` / `syntaxes:` / `fonts:` frontmatter can point at a file or a directory: layouts/css/syntaxes read `*.html` / `*.css` / `*.sublime-syntax` in filename order, while `fonts:` copies files without an extension filter so `.woff2`, `.ttf`, and `@font-face` CSS can live side by side. When omitted, auto-detects `layouts/` / `css/` / `syntaxes/` / `fonts/` **next to the deck** (zero-config convention, adopted per Issue #17; falls back to the built-in default for layouts/css/syntaxes if absent). Fonts are copied verbatim into build output, preview cache, present cache, and PDF export workspace by file/directory, not by extension. CSS validation is uniform across all files: keyed selectors are checked against the slots of that slide's layout, and bare `.slot-*` are checked against the union of provided layouts
docs/plans/           Implementation plans for each milestone (history)
```

Docs site https://peitho.gosu.ke/ (Cloudflare Pages project `peitho`): the Zola docs site lives in `site/` (landing, guide with a left sidebar, examples gallery) and the built decks live under `/demo/<name>/`. `make demo-site` first runs `make docs-sources` (copies each `$(DEMO_DECKS)` deck's `deck.md` into `site/static/deck-sources/` so example pages can show the source), then `zola build` for `site/` into `.demo-site/` and `peitho build` for every deck in `$(DEMO_DECKS)` into `.demo-site/demo/<name>/`; the Zola output is scanned for `page(s) ignored` so a weightless page in a `sort_by="weight"` section fails the build (Zola drops it silently otherwise). `make demo-screenshots` renders each demo deck's first slide with Playwright chromium into `.demo-site/deck-shots/` (gallery tiles / per-example OGP images), and `make og-cards` renders the brand and per-section OGP cards. `.github/workflows/deploy-demo.yml` installs Zola 0.22.1, Node 22, and Playwright chromium, then runs `make demo-site && make demo-screenshots && make og-cards` on every `main` push and deploys to Cloudflare Pages; same-repo PRs get a preview deploy at `https://<sanitized-branch>.peitho-4yr.pages.dev` (Cloudflare truncates the alias to 28 characters and strips surrounding hyphens) with a PR comment, cleaned up on close by `demo-preview-cleanup.yml`. PR builds pass `ZOLA_BASE_URL="https://<alias>.peitho-4yr.pages.dev"` so absolute URLs in the preview point at the preview host, not production. Manual fallback: `envchain peitho make deploy-demo`. Adding an example requires adding it to `$(DEMO_DECKS)` in `Makefile` (all demo recipes loop over that list) and adding `site/content/examples/<name>.md` — the landing page's Examples gallery derives from the examples section at build time (`gallery_from = "examples"` in `site/content/_index.md`, Issue #274), so there is no third wiring site; `site/static/_redirects` is a frozen eight-line legacy list for the pre-move deck URLs and never grows. Design record: `docs/plans/2026-07-09-demo-site-deploy-flow.md` and `docs/plans/2026-07-09-docs-site.md`.

## Pitfalls (facts confirmed by measurement — no need to re-investigate)

- **SSE does not work with tiny_http**: when data_length is None, bodies below a threshold are buffered until EOF, and the chunk encoder doesn't flush small chunks either. That's why /sync uses long polling (`GET /sync?seq=N`; a GET with no query returns immediately = join handshake; every JSON GET response (handshake and 200 poll; the 30s poll timeout is an empty 204) carries `{"seq":…,"message":…,"index":…,"swapped":…,"generation":…}` — current absolute state on top of the latest message; `POST /sync` only accepts `{index}|{swapped}|{close:true}` and generation is server-internal)
- **Flag handoff to an already-running Chrome instance only works via `--app`**: `--start-fullscreen`/`--window-position`/`--window-size` are ignored. To reliably apply them, launch a new process with a separate `--user-data-dir` (which is why slides/presenter use two instances, `~/.peitho/chrome-profile-{slides,presenter}`)
- **PDF export flattens gradients at print time**: `pdf.html` embeds a script that converts CSS gradients to Chrome-rasterized images because Type 4 shadings render pathologically slowly and incorrectly in macOS Preview.app (measured 2026-07-06). Design record: `docs/plans/2026-07-06-pdf-gradient-flatten.md`
- **PDF export flattens box-shadow at print time**: Chrome emits blurred CSS box-shadows as `/S /Luminosity` soft masks that Quartz renders as hard black rectangles, so `pdf_flatten.js` rasterizes supported shadows to RGBA PNGs for PDF export (measured 2026-07-06). Design record: `docs/plans/2026-07-06-pdf-shadow-flatten.md`
- **`image.decode()` hangs forever (neither resolves nor rejects) in Linux headless Chrome under `--virtual-time-budget`** (measured 2026-07-06, Issue #155): no rejection means `.catch()` never fires — the hang is completely silent, and Chrome prints the page once the virtual-time budget expires. On macOS decode just happened to finish before the budget ran out. In-page scripts run under virtual time (`pdf_flatten.js`) therefore never rely on `decode()` and wait on load/error events instead; reintroduction is guarded by the Linux `e2e` CI job (with `PEITHO_CHROME_PATH` set explicitly so a missing Chrome panics instead of silently skipping). Design record: `docs/plans/2026-07-06-pdf-flatten-linux-decode-hang.md`
- **On macOS, Chrome's process lingers even after all windows are closed**: if the previous present instance is still holding the peitho profile, the next launch becomes a handoff and all placement flags are lost. That's why present terminates any lingering process before opening
- **Lingering Chrome processes must be terminated normally, not killed with SIGTERM**: SIGTERM registers as a crash to Chrome (`exit_type: Crashed`), triggering crash recovery on the next launch which restores the old session's windows/bounds. Using `NSRunningApplication.terminate` (in JXA this fires via **parenthesis-less** access, due to the bridge — confirmed by testing) terminates it as Normal. Note that `exit_type` always shows Crashed while running (it's written up front at launch, then flipped back to Normal on clean exit), so reading it while the process is active is meaningless. Restrict target pids to the Chrome main process (no `--type=`) from `ps` (`pgrep -f` also catches the shell itself if it contains the pattern)
- **Chrome 149 on macOS may never exit after one-shot headless PDF export**: `--print-to-pdf` can complete its output but keep the parent process alive after waking GoogleUpdater. Export code must use the shared one-shot runner (spawn + piped readers + completion predicate + timeout + kill/reap), never `Command::output()`. Killing is acceptable here because export uses throwaway temp `--user-data-dir` profiles; the SIGTERM crash-recovery warning above applies to persistent presenter profiles
- **Chrome's `--app` window position restoration breaks if the app name contains a dot**: the position is saved under `browser.app_window_placement` keyed by a URL-derived app name (host+path), but a dot in the name (as in `127.0.0.1` or `.html`) gets expanded as a pref path when written, causing a mismatch on read and preventing restoration (confirmed Chromium behavior, measured). That's why presenter opens at `http://localhost:<port>/presenter` (an extensionless route). Since the app name doesn't include the port, restoration still works even though the port changes each time
- **On the first `--app` launch with no placement flags, which display the window appears on is undetermined**: if it appears on the slides-side display, it can end up completely hidden behind the fullscreen Space (it won't even show up in CGWindowList's OnScreenOnly). That's why windowed mode only seeds `--window-position`+`--window-size` to center on the primary display when there's no saved placement or the saved position is off-screen; when there is a visible saved placement, it's left flag-free and Chrome's own restoration is trusted (the check is peitho reading the profile's Preferences). Chrome clamps partially off-screen saved bounds at launch
- **BroadcastChannel does not reach across different profiles**: that's why sync goes through the server (a deliberate extension from §15; the layering — DOM events bridged to a transport — remains unchanged)
- **A CLI-launched app window is closed via `window.close()`** (because it has only 1 history entry). Esc triggers `peitho:closerequest` → `{close:true}` broadcast to all windows → each closes itself → the server also unblocks and exits after a grace period
- **requestFullscreen/window.open require transient user activation**: awaiting a permission prompt in between invalidates it. In-browser window placement failed twice this way before the switch to CLI-driven placement (M8/M9/M10)
- NSScreen has a bottom-left origin. Chrome's `--window-position` uses top-left. The conversion lives in `displays.rs` (pure functions + tests against measured values)
- In vitest tests, always destroy/clean up the shell/listeners (listener contamination on the shared window causes multiple firings)
- `.peitho/present-cache/` is recreated every time (the adopted value from the §18 cache policy). `.peitho/preview-cache/` contains generation directories and swaps between them after emit completes, keeping only the current and previous generations. `dist/slides/` is also cleared on every build (to prevent stale fragments from leaking into publish)
- **pulldown-cmark 0.10 loops forever on indented `---` dense pairs once `ENABLE_YAML_STYLE_METADATA_BLOCKS` is on** (upstream #912, fixed in 0.11.1) — that is why the workspace pins 0.13. **0.13 still tokenizes dense mid-document `---` pairs as metadata blocks** (measured; legitimate slides like `---`/`# Cfg`/`port: 8080`/`---` get swallowed), so frontmatter detection and slide splitting use two grammars: metadata-enabled only for the leading block, the legacy grammar for splitting. Do not load metadata blocks into the split grammar
- **A named ES-module import from a bundle that lacks the export fails at module instantiation** — showError never runs and the window is just black (a recurring incident class). The present/presenter entry pages therefore use a namespace import (`import * as peitho from './shell.js'`) plus feature detection, so a stale `--shell` bundle degrades to a visible error instead
- Explicit slot syntax (2026-07-05, Issue #21): when convention mapping (title/code/image/body) can't tell where content belongs — two-column layouts, multiple `blocks` slots — the author writes `::: {slot=name}` fenced blocks. The parser tokenizes them line-first (fenced-code interior excluded), produces a `FragmentKind::SlotGroup { name: ExplicitSlot, children }` in Parsed, and mapping expands the children into the named slot; SlotGroup never leaves the Mapped phase. `ExplicitSlot`'s constructor is `pub(crate)` so no consumer outside the parser can fabricate one — this preserves the invariant that convention- and author-declared routing cannot be silently interchanged. Slidev-style `<template #name>` was rejected because it would leak HTML component vocabulary into content (pillar ①). Validation is exhaustive (11 rules, all line-numbered errors): unclosed/nested/empty divs, 4+ colon fences, missing/unknown/multi attributes, invalid slot names, unknown slot names against the layout, accepts violations via existing check.rs. Design record: `docs/specs/2026-07-05-explicit-slot-syntax.md`, implementation plan: `docs/plans/2026-07-05-explicit-slot-syntax.md`
